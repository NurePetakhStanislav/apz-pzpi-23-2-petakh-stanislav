3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Загальні відомості про систему

Назва системи: TikTok.
Розробник: ByteDance.
Призначення системи: TikTok — це платформа для створення та перегляду коротких відео. Система дозволяє користувачам публікувати контент, взаємодіяти з іншими користувачами, переглядати персоналізовану стрічку відео та використовувати алгоритмічні рекомендації для підбору контенту.

3.2 Архітектурний стиль системи

Архітектурний стиль даної системи є мікросервісна архітектура з елементами подієво-орієнтованої системи. TikTok складається з великої кількості незалежних сервісів, які відповідають за різні функції: рекомендації, обробку відео, авторизацію, взаємодію з користувачами. Комунікація між сервісами відбувається через API та черги повідомлень.

3.3 Основні компоненти системи

Інтерфейс користувача:
- стрічка відео;
- профіль каналу;
- рядок пошуку;
- особистий чат.
Серверна частина:
- API Gateway;
- сервіс користувачів;
- сервіс відео;
- сервіс рекомендацій.
Рівень бізнес-логіки:
- алгоритм рекомендацій контенту;
- обробка взаємодій (лайки, коментарі);
- модерація контенту;
- формування рейтингу відео.
Рівень зберігання даних:
- відео;
- профілі користувачів;
- історія переглядів;
- взаємодії.
Допоміжні сервіси:
- система рекомендацій;
- збереження відео;
- кешування;
- фонова обробка відео;
- аналітика.

3.4 Взаємодія компонентів

Кожна дія всередині системи виглядає наступним чином:
- Користувач відкриває додаток;
- Клієнт надсилає запит на сервер;
- API Gateway перенаправляє запит до сервісів;
- Сервіс рекомендацій формує стрічку відео;
- Відео завантажується через CDN;
- Користувач взаємодіє з контентом.

UML-діаграма взаємодії виглядає наступним чином (див. рис. 2. 1).

Рисунок 2. 1 — UML-діаграма взаємодії компонентів Tik-Tok
Клієнт взаємодіє з API Gateway, який розподіляє запити між мікросервісами.
Користувацький сервіс відповідає за реєстрацію та авторизацію, профіль користувача, редагування даних та зберігання даних про підписки. Вся інформація про користувачів зберігається та редагується в базі даних, а кеш дозволяє швидко відображати свій профіль. Сервіс контенту завантажує, видаляє чи зберігає відео в сховище файлів та його метадані в базу даних. Сервіс взаємодії відповідає за лайки, коментарі, підписки та репости. Кожна взаємодія зберігається в базі даних та обробляється в алгоритмах. Сервіс рекомендацій підбирає відео, аналізує інтереси, створює стрічку на основі смаків та рекомендацій. Кеш дозволяє зберігати стрічку відео без частого звертання до бази даних.

3.5 Масштабованість та надійність

Балансування навантаження:
Запити користувачів розподіляються між серверами через load balancer, щоб не перевантажувати один.
Відмовостійкість:
Архітектура забезпечує роботу системи при відмові окремих сервісів завдяки їх незалежності та резервуванню.
Кешування:
Використовується для зберігання часто запитуваних даних (стрічка, профілі, популярний контент), що зменшує навантаження на базу даних.
Реплікація:
Дані зберігаються у кількох копіях бази даних для забезпечення надійності та доступності.
Контейнеризація або хмарна інфраструктура:
Мікросервіси розгортаються у контейнерах та можуть масштабуватися в хмарному середовищі.
